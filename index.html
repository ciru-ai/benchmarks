Setup and caveats
Machine details, runtime settings, and the benchmark notes that meaningfully affect interpretation.
+

System

  • Host ciru, NixOS, Linux 7.0.1
  • AMD Ryzen AI Max+ 395, 16 cores / 32 threads
  • AMD Radeon 8060S-class Strix Halo iGPU, RADV STRIX_HALO
  • 128 GiB total UMA memory, benchmarked with large GPU aperture available
  • Reported pools include 64 GiB VRAM and about 31.2 GiB GTT on the tested setup

Runtime

  • llama.cpp with Vulkan, plus isolated ROCm comparisons where noted
  • Current system benchmark build 8864; b8672 used for refreshed Gemma 4 rows; upstream b8971 used for checkpointed speculative tests
  • Common benchmark flags -ngl 999 -fa 1 -n 128 -o md, usually -r 1 or -r 2
  • Guard reserves: 8 GiB RAM, 2 GiB VRAM, 4 GiB GTT
  • Typical contexts: 4096, 16384, 32768, with long checks to 100000+

Important notes

  • Some later April tables report only peak GPU memory; those points use that value as GPU footprint and leave system RAM/GTT blank.
  • Speculative decoding points are separated from the llama-bench frontier because active prompt occupancy changes the meaning of the context number.
  • Qwen3.6 MTP rows in the KV/backend lab are server-side multi-turn chat timings from mtpbench.md, not clean llama-bench context rows.
  • Dynamic Strix quant rows are clean prompt/decode checks from the MTP workspace and are included in the main context charts.
  • Gemma 4 E2B 16K uses the refreshed b8672 f16 KV row from 2026-05-07; the older 2026-04-02 16K row was superseded.
  • Gemma 4 MTP server tests are recorded in the history, but clean llama-bench rows remain the chart source for context-planning comparisons.
  • Gemma 4 31B Dense was manually stopped at 32768, so it is excluded beyond measured contexts.
  • Qwen Coder Next is plotted as 80B-class on the parameter axis to match the benchmark notes in this repo.
