System

The machine profile that matters for these benchmark runs.

  • AMD Ryzen AI Max+ 395 with 16 cores / 32 threads.
  • AMD Radeon 8060S-class Strix Halo iGPU using unified memory.
  • 128 GiB total RAM, currently configured as a 64/64 split for system and graphics use.
  • Benchmark tooling reports roughly 62 GiB system RAM, 64 GiB VRAM pool, and 31.2 GiB GTT pool.
  • NixOS on Linux 6.19.9; logs and models live under /srv/llm/....
  • No swap; peak RAM, VRAM, and GTT were sampled during each run.

Runtime

Shared benchmark settings so the model comparisons stay apples-to-apples.

  • Backend: llama.cpp with Vulkan via llama-cpp-vulkan.
  • Pinned commit: 08f21453aec846867b39878500d725a05bd32683.
  • Primary wrapper: machine-setup/bench-llama.sh.
  • Default flags: -ngl 999 -fa 1 -n 128 -r 2 -o md.
  • Typical contexts: 4096, 16384, 32768, with long-context checks up to 100000 and beyond.
  • Memory guard: llm-guard reserves 8 GiB RAM, 2 GiB VRAM, and 4 GiB GTT.

Notes & Caveats

The details that matter when you read the charts.

  • The Gemma 4 runs on 2026-04-02 used a newer local Vulkan-enabled llama.cpp build because the older system build failed with unknown model architecture: 'gemma4'.
  • The dense Gemma 4 31B run was manually stopped at 32768 after more than ten minutes, so that data point is shown as aborted rather than compared numerically.
  • The page focuses on benchmark tables with saved manifests and per-context logs. Partial tuning checks and older hand-entered notes remain in the markdown history but are not mixed into the main comparison charts.
  • The most stable comparison lane on this machine remains 4096, 16384, and 32768; long-context probes are included when they exist, but not every model was pushed that far.
  • Qwen3-Coder-30B-A3B is the standout “large but still fast” coding-class result, while Gemma 4 E2B is the outright throughput winner in the newest wave.

Source of truth is still the markdown log. This page just gives it a cleaner cockpit.
